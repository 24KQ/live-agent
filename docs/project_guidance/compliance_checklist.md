# LiveAgent 合规与安全检查表

> **版本**: v1.0  
> **更新日期**: 2026-07-07  
> **用途**: 每次新增 Tool、数据表、事件流、外部接口或演示能力前，必须用本文档做工程合规自查。  
> **适用范围**: 本地开源复刻版、MVP、演示环境、后续真实平台适配前的预审。

---

## 1. 总原则

本项目允许做本地模拟、脱敏样例、工程验证；不允许在未经过平台授权和正式合规评审前连接真实淘宝生产 API、真实交易、真实订单、真实观众个人信息。

核心原则：

- **公开仓库不记录真实凭据**：文档只保留端口、用途和环境变量名，真实账号密码只能存在于开发者本地 `.env` 或部署平台 Secret。
- **高危操作必须审批**：改价、发券、批量上架、批量下架等操作必须 hard-gate 或 soft-gate。
- **状态必须可追溯**：任何业务状态变化必须能追溯到 `trace_id`、`tool_call_id` 或 `approval_id`。
- **默认拒绝**：未知工具、未知阶段、未知风险等级、缺少幂等键的写操作，一律 fail-closed。

## 2. 基础设施合规

| 检查项 | 要求 | 状态 |
| :--- | :--- | :---: |
| PostgreSQL 连接 | 仅使用本地 `localhost:5432`，账号密码从 `${POSTGRES_USER}` / `${POSTGRES_PASSWORD}` 注入 | 待检查 |
| Redis 连接 | 仅使用本地 `localhost:6379`，无密码 | 待检查 |
| Kafka 连接 | 仅使用本地 `localhost:9092` | 待检查 |
| MinIO 连接 | 如启用，仅使用本地 `localhost:8900`，账号密码从 `${MINIO_ACCESS_KEY}` / `${MINIO_SECRET_KEY}` 注入 | 待检查 |
| pgvector | 首次使用前执行 `CREATE EXTENSION IF NOT EXISTS vector;` | 待检查 |
| Kafka KRaft | 首次消费者组使用前确认 `__consumer_offsets` topic 可用 | 待检查 |
| 密钥管理 | `.env` 不提交，公开文档只保留变量名和占位符 | 待检查 |

## 3. 数据合规

| 检查项 | 要求 | 阻断条件 |
| :--- | :--- | :--- |
| 示例数据 | 必须使用模拟主播、模拟商品、模拟弹幕 | 出现真实手机号、订单号、身份证、真实用户 ID |
| 弹幕数据 | MVP 只允许模拟弹幕或脱敏文本 | 接入真实直播弹幕且未说明授权来源 |
| 记忆内容 | L1 主观偏好必须来自主播确认 | 未确认的推断偏好直接写入 L1 |
| 审计日志 | 工具调用、审批、状态变更必须记录 | 高危操作无审计记录 |
| 数据保留 | 默认本地保留 30 天，可配置 | 永久保留且无清理策略 |
| 导出报告 | 不包含真实个人信息 | 报告中泄露真实观众身份 |

## 4. Tool 新增检查表

新增任何 Tool 前，必须填写以下内容：

| 字段 | 要求 | 示例 |
| :--- | :--- | :--- |
| 工具名 | 动词短语，唯一且可读 | `change_price` |
| 生命周期 | 明确允许阶段 | `["ON_LIVE"]` |
| 风险等级 | auto / soft-gate / hard-gate / block | `hard-gate` |
| 参数 Schema | 必须有类型、必填项、范围约束 | `new_price >= 0` |
| 返回 Schema | 必须结构化 | `{status, result, error_code}` |
| 幂等键 | 有副作用工具必须要求 | `idempotency_key` |
| 审批策略 | hard-gate 必须记录审批 | `approval_log` |
| 审计策略 | 所有工具必须写 audit | `tool_call_audit` |
| 失败策略 | 明确是否重试、降级、阻断 | 参数错修复一次，系统错退避重试 |

Tool 上线阻断条件：

- 未注册生命周期。
- 未注册风险等级。
- 有副作用但没有幂等键。
- hard-gate 工具没有审批记录。
- 直接让 LLM 写数据库。
- 参数 Schema 不限制价格、库存、批量数量等关键字段。

---

## 5. 生命周期合规

| 阶段 | 允许 | 禁止 |
| :--- | :--- | :--- |
| PRE_LIVE | 查询货盘、排品、生成手卡、模拟建播、偏好对账 | 播中改价、发券、真实切品 |
| ON_LIVE | 查询、切品、改价、发券、弹幕回复、售罄下架 | 未审批的大幅降价、负数价格、越权写记忆 |
| POST_LIVE | 数据归因、决策复盘、信任分更新、记忆回写 | 改价、发券、切品、建播 |

验证要求：

- POST_LIVE 调用 `change_price` 必须被 block。
- PRE_LIVE 调用 `issue_coupon` 必须被 block。
- trust_score < 0.4 时，高危写工具必须从 LLM 可见工具列表中移除。

---

## 6. 审批与幂等检查

| 检查项 | 要求 |
| :--- | :--- |
| soft-gate | 记录提示内容、工具名、参数、时间 |
| hard-gate | 等待主播确认，记录通过/拒绝/过期 |
| block | 直接拒绝并记录原因 |
| idempotency_key | 同一个 key 重复调用返回首次结果，不重复执行 |
| 审批超时 | MVP 可不设超时；后续 Web 副屏应支持过期状态 |

---

## 7. 输出内容检查

Agent 给主播的输出必须满足：

- 引用商品信息时，商品 ID 必须存在于当前 State。
- 价格、库存、优惠券必须来自当前 State 或工具返回，不能编造。
- 信任分低于 0.4 时，只输出证据，不输出强建议。
- 涉及平台规则、风控词、禁用词时，必须标注“本地模拟规则”或引用规则来源。
- 播中输出要短，优先展示“结论、原因、操作按钮”。
