## 1. 文档目标与项目边界

### 1.1 文档目标

本文档用于指导 LiveAgent（淘宝主播 Agent 开源复刻版）从 0 到 1 落地。它不是营销介绍，也不是只给人看的概念说明，而是后续研发、测试、合规自查、AI 编码 Agent 执行任务时共同遵循的产品基线。

本文档重点回答四个问题：

| 问题 | 本文档回答方式 |
| :--- | :--- |
| 为什么做 | 说明直播场景中主播注意力稀缺、运营决策高频、突发事件高风险等痛点 |
| 做什么 | 定义播前、播中、播后全生命周期功能范围 |
| 先做什么 | 定义 MVP 与 Phase 1-4 分期优先级 |
| 怎么算完成 | 为 P0/P1 功能补充触发条件、输入输出、验收标准与失败处理 |

### 1.2 项目定位

LiveAgent 是淘宝直播 AI 主播助理的**开源复刻版原型系统**，用于本地研发与技术验证。

一句话定义：LiveAgent 是一个能在直播全生命周期（播前策划 -> 播中实时响应 -> 播后复盘）中，持续为主播提供运营决策辅助的 AI Agent。它不是虚拟主播，也不是普通聊天机器人，而是一个有状态、有记忆、有安全边界、能自我纠偏的智能副驾驶。

### 1.3 明确边界

| 类型 | 范围 |
| :--- | :--- |
| 当前要做 | 本地模拟直播间、本地商品货盘、模拟弹幕事件、模拟库存事件、模拟主播采纳/拒绝反馈 |
| 当前不做 | 真实淘宝开放平台接入、真实资金交易、真实用户隐私数据采集、真实直播间自动操作 |
| 预留能力 | 后续可替换模拟网关为真实平台适配器，可将副屏从终端升级为 Web UI |
| 安全底线 | 所有高危写操作必须经过框架层拦截，不允许只靠 Prompt 约束 |

### 1.4 非目标

以下内容不属于本项目第一阶段目标：

| 非目标 | 原因 |
| :--- | :--- |
| 直接接入真实淘宝生产 API | 需要平台授权、合规审查和真实资金风险控制 |
| 自动修改真实商品价格 | 存在资金损失风险，开源复刻版只做模拟 Reducer |
| 处理真实观众个人信息 | 本地演示只使用模拟弹幕与脱敏样例 |
| 训练专属大模型 | 第一阶段依赖通用 LLM 与 Harness 工程，不做模型训练 |
| 完整商家后台 | 本项目聚焦 Agent Harness，不做大型电商管理系统 |

## 2. 业务背景与核心价值

### 2.1 业务背景

淘宝直播间的主播在一场 4-8 小时的直播中，需要同时处理以下事务：

- 讲解 30-100 个商品，每个商品有不同卖点、价格策略和话术。
- 实时阅读弹幕并回复观众问题。
- 根据流量变化临时调整排品顺序。
- 处理突发事件，例如爆款秒空、平台合规警告、库存变动。
- 在直播结束后复盘数据、总结经验、更新下一场策略。

核心痛点：主播的注意力是极度稀缺资源。在高强度、高并发的直播场景下，主播无法靠人力同时兼顾讲解、互动和运营决策，需要一个实时在线的 AI 副驾驶来分担运营侧压力。

### 2.2 核心价值主张

| 价值 | 说明 | MVP 衡量方式 |
| :--- | :--- | :--- |
| 减负 | 将查数据、算价格、排商品等运营杂务交给 Agent 辅助 | 播前能自动生成可读排品方案和商品手卡 |
| 提效 | 将播前准备流程结构化、自动化 | 10 个商品内的排品与手卡生成可在一次流程中完成 |
| 防灾 | 对售罄、库存低、价格越界等风险做即时拦截 | 售罄事件能在 5 秒内进入应急处理流程 |
| 可控 | 高危工具不由模型直接执行 | 改价、发券等操作必须经过 hard-gate |
| 越用越懂 | 跨场次保存偏好、行为和信任分 | 下一场启动时能加载历史偏好和 trust_score |

## 3. 用户角色与使用方式

### 3.1 核心用户

| 角色 | 说明 |
| :--- | :--- |
| 主播 | 直播间真人主播，在副屏查看 Agent 建议，并通过按钮、终端命令或模拟输入采纳/拒绝 |
| 运营/中控 | 可在演示环境中模拟库存变动、弹幕流、流量变化和平台告警 |
| 开发者 | 使用本文档和 Design Spec 实现本地复刻版，验证 Harness 架构 |

### 3.2 主播交互原则

- 主播不需要理解 Agent 内部架构。
- 主播只需要看到“建议、原因、风险、可选动作”。
- 高危动作必须让主播明确确认。
- 信任分低时，Agent 应减少主观建议，只展示客观数据。
- 播中交互必须短、稳、可扫读，不能长篇解释。

## 4. 端到端业务流程

### 4.1 播前 (PRE_LIVE)

目标：帮助主播完成选品、排品、手卡生成、偏好对账和模拟建播。

标准流程：

1. 主播创建或选择一场直播会话。
2. Agent 加载本场货盘、历史偏好、历史转化数据和 trust_score。
3. Agent 生成推荐排品方案。
4. Agent 为商品生成讲解手卡。
5. 若发现 L1 偏好与 L3 行为矛盾，触发偏好对账确认。
6. 主播确认后执行模拟建播。

### 4.2 播中 (ON_LIVE)

目标：在直播过程中实时响应弹幕、库存、流量和主播临时指令。

标准流程：

1. 生命周期切换到 `ON_LIVE` 后，系统只开放播中工具。
2. Kafka 或模拟事件源持续产生弹幕、库存和流量事件。
3. Gateway 对事件做 5 秒滑窗聚合和优先级分级。
4. P0/P1 事件进入优先级队列。
5. PlanEngine 根据事件生成应对动作。
6. PreToolCall Hook 做工具风险拦截。
7. Reducer 执行确定性状态更新。
8. 副屏或终端展示建议、状态和审计结果。

### 4.3 播后 (POST_LIVE)

目标：复盘本场决策、更新信任分、回写记忆，并锁定业务写操作。

标准流程：

1. 生命周期切换到 `POST_LIVE` 后，禁用改价、切品、发券等业务写操作。
2. Agent 汇总本场商品表现、建议采纳情况和业务结果。
3. 系统生成 Decision Trace 复盘。
4. TrustManager 计算 trust_score 变化。
5. MemoryStore 写入经过确认的行为记忆。
6. 生成播后报告。

## 5. MVP 范围

### 5.1 MVP 必须完成

| 编号 | 能力 | 验收口径 |
| :--- | :--- | :--- |
| MVP-01 | 生命周期状态机 | 能在 `PRE_LIVE -> ON_LIVE -> POST_LIVE -> PRE_LIVE` 间合法切换，并拒绝非法跳转 |
| MVP-02 | 商品货盘状态 | 能创建、查询、更新本地模拟商品，价格、库存、上架状态由 Reducer 管理 |
| MVP-03 | 工具注册表 | 每个工具有生命周期、风险等级、参数 Schema 和说明 |
| MVP-04 | 四级安全拦截 | auto、soft-gate、hard-gate、block 四类策略可测试 |
| MVP-05 | 播前排品与手卡 | 能基于模拟货盘生成排品方案和商品手卡 |
| MVP-06 | 播中售罄应急 | 售罄事件能触发自动下架、推荐备选商品和副屏提示 |
| MVP-07 | 决策审计 | 每次建议、采纳/拒绝、工具调用均有日志 |
| MVP-08 | 信任分 | 播后能基于 Decision Trace 更新 trust_score |
| MVP-09 | 本地中间件接入 | PostgreSQL、Redis、Kafka 至少完成本地连接验证 |

### 5.2 MVP 可以模拟

| 模拟对象 | 模拟方式 |
| :--- | :--- |
| 直播间弹幕 | 本地脚本向 Kafka topic 写入模拟弹幕 |
| 库存变化 | 本地脚本或测试用例写入库存事件 |
| 主播确认 | CLI 输入或 Web 副屏按钮模拟 |
| 商品数据 | PostgreSQL 初始化脚本写入样例商品 |
| 业务效果 | 播后通过模拟 conversion_rate、GMV、lift 计算 |

## 6. 功能需求清单

### 6.1 播前功能

| 编号 | 名称 | 优先级 | 触发条件 | 输入 | 输出 | 验收标准 | 失败处理 |
| :--- | :--- | :---: | :--- | :--- | :--- | :--- | :--- |
| F-01 | 货盘查询 | P0 | 主播进入播前或手动查询 | room_id、筛选条件 | 商品列表、库存、价格、转化率 | 查询结果来自状态库，不由 LLM 编造 | DB 不可用时返回只读错误并阻断后续排品 |
| F-02 | 智能排品方案 | P0 | 主播请求生成开播方案 | 商品列表、主播偏好、历史转化 | 排品顺序、商品定位、推荐理由 | 输出包含引流款/利润款/氛围款，且每个商品 ID 可回溯 | LLM 输出不合 Schema 时重试一次，仍失败则返回模板方案 |
| F-03 | 商品手卡生成 | P0 | 排品方案确认或主播指定商品 | product_id、商品信息、风控词 | 卖点、话术、价格建议、禁用词提醒 | 不允许出现商品库不存在的信息 | 商品缺失时跳过该商品并记录失败 |
| F-04 | 一键建播 | P1 | 主播确认播前方案 | room_id、排品方案、初始价格 | 模拟直播间配置结果 | 只能在 PRE_LIVE 执行，执行前 soft-gate 提示 | 被拒绝时保留方案，不执行写操作 |
| F-05 | 偏好对账确认 | P1 | L1 偏好与 L3 行为矛盾累积超阈值 | 旧偏好、行为证据、推荐新偏好 | 确认问题、主播选择、记忆更新结果 | 主播未确认前不得覆盖 L1 偏好 | 主播拒绝时重置 evidence_weight |

### 6.2 播中功能

| 编号 | 名称 | 优先级 | 触发条件 | 输入 | 输出 | 验收标准 | 失败处理 |
| :--- | :--- | :---: | :--- | :--- | :--- | :--- | :--- |
| F-06 | 弹幕实时响应 | P0 | Kafka 收到弹幕事件 | 弹幕文本、商品状态、风控词 | 聚合问题、参考回复 | 5 秒窗口内相似问题合并，回复只作参考 | LLM 超时时返回兜底话术 |
| F-07 | 库存预警与自动下架 | P0 | 商品库存为 0 或售罄事件 | product_id、库存事件 | 下架 Action、备选商品建议、预警 | 5 秒内完成状态更新和提示 | Reducer 写入失败时进入人工接管状态 |
| F-08 | 智能切品建议 | P1 | 流量激增/骤降或主播请求 | viewer_count、当前商品、排品队列 | 推荐切换商品与理由 | 只建议可上架且有库存商品 | 无备选商品时只提示风险 |
| F-09 | 改价与发券 | P1 | 转化率低或主播请求促销 | product_id、建议价格、券规则 | 改价/发券建议或模拟执行结果 | 降幅超过阈值必须 hard-gate | 主播拒绝后触发 Replan，不执行写操作 |
| F-10 | 临时计划调整 | P1 | 主播跳过、插播、暂停商品 | 当前 DAG、主播指令 | 更新后的计划 DAG | 已完成节点不重复执行，未受影响节点保留 | 指令无法解析时请求澄清 |
| F-11 | 紧急事件抢占 | P0 | Level 5 事件到达 | 当前任务、紧急事件 | 冻结 checkpoint、应急处理结果、恢复任务 | 低优任务可冻结和恢复 | 恢复失败时保留 checkpoint 并提示人工处理 |

### 6.3 播后功能

| 编号 | 名称 | 优先级 | 触发条件 | 输入 | 输出 | 验收标准 | 失败处理 |
| :--- | :--- | :---: | :--- | :--- | :--- | :--- | :--- |
| F-12 | 直播数据归因 | P0 | 生命周期进入 POST_LIVE | 本场商品数据、事件日志 | GMV/转化率/lift 分析 | 报告包含每个核心商品表现 | 数据缺失时标注不可归因 |
| F-13 | 决策复盘 | P1 | 播后复盘启动 | decision_trace_log | 建议准确率、采纳率、错误案例 | 每条建议可追溯到 trace_id | trace 缺失时写入审计异常 |
| F-14 | 信任度更新 | P1 | 决策复盘完成 | trust_score、trace、lift | 新 trust_score、trust_delta | 分数钳制在 0.0-1.0 | 计算失败时保持旧分数并记录错误 |
| F-15 | 记忆回写 | P0 | 主播确认复盘或自动归纳完成 | 行为摘要、主播反馈 | L1/L2/L3 记忆条目 | 业务写操作锁定不影响记忆/审计写入 | 未确认的主观偏好不得写入 L1 |
| F-16 | 播后写操作锁定 | P0 | 进入 POST_LIVE | lifecycle、tool_name | 工具掩码结果 | 改价/切品/发券全部不可见且不可执行 | 幻觉调用被 fail-closed 拦截 |

## 7. 非功能需求与约束

### 7.1 安全约束

| 编号 | 名称 | 描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| NF-01 | 分级安全拦截 | 工具调用前必须经过 auto、soft-gate、hard-gate、block 风险评级 | 每个工具有风险等级，测试覆盖四级分支 |
| NF-02 | 写操作生命周期锁定 | 业务写操作只能在允许阶段出现 | POST_LIVE 调用改价必须被拒绝 |
| NF-03 | 信任驱动能力降级 | trust_score 低时物理移除高危工具 | trust_score < 0.4 时只保留只读/建议类工具 |
| NF-04 | 幂等保护 | 有副作用工具必须携带 idempotency_key | 重复请求不会重复改价、切品、发券 |
| NF-05 | 审批留痕 | hard-gate 必须记录审批人、时间、参数和结果 | 审计表可按 approval_id 回放 |

### 7.2 可靠性约束

| 编号 | 名称 | 描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| NF-06 | 断点续传 | 崩溃或人为中断后从 checkpoint 恢复 | 手卡生成被中断后能继续未完成节点 |
| NF-07 | 状态确定性 | 商品价格、库存、上架状态只能由 Reducer 更新 | LLM 不能直接写 DB |
| NF-08 | 上下文隔离 | 重型生成任务在子任务中执行 | 主控上下文不保留完整手卡生成中间过程 |
| NF-09 | Fail-Closed | 不确定是否允许时默认拒绝 | 未注册工具调用一律 block |

### 7.3 性能约束

| 编号 | 名称 | 描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| NF-10 | 紧急响应时间 | 售罄等 P0 事件从产生到提示不超过 5 秒 | 集成测试中 P95 <= 5 秒 |
| NF-11 | 弹幕聚合窗口 | 弹幕按 5 秒滑动窗口聚合去重 | 同义高频问题只触发一次回复 |
| NF-12 | Token 预算 | 单轮推理只注入必要 state 与摘要 | 不把完整历史工具 JSON 追加到 messages |

### 7.4 数据与合规约束

| 编号 | 名称 | 描述 | 验收标准 |
| :--- | :--- | :--- | :--- |
| NF-13 | 本地数据边界 | MVP 使用模拟数据和脱敏样例，不采集真实观众个人信息 | 示例数据不包含手机号、身份证、真实订单号 |
| NF-14 | 配置与密钥边界 | 公开仓库不记录真实本地密码，开发和生产都必须通过环境变量/密钥管理注入 | 文档只保留占位符和变量名 |
| NF-15 | 决策可审计 | Agent 推荐、主播反馈、工具调用、审批均可追踪 | 任一业务状态变更可追溯到 trace_id |
| NF-16 | 数据保留 | 日志和记忆要有保留策略 | MVP 默认保留 30 天，可配置 |

## 8. 本地基础设施依赖

本项目开发环境依赖本地 Docker 中间件。公开仓库只记录端口、用途和环境变量名，不记录真实密码；开发者应从自己的 Docker Compose 配置或本机 `D:\Others\DockerData\PORTS.md` 查询真实值，并写入本地 `.env`。

| 服务 | 地址 | 端口 | 账号 | 密码 | 本项目用途 | 是否必需 |
| :--- | :--- | :--- | :--- | :--- | :--- | :---: |
| PostgreSQL + pgvector | localhost | 5432 | `${POSTGRES_USER}` | `${POSTGRES_PASSWORD}` | 主状态库、记忆库、审计库、checkpoint | 是 |
| pgAdmin | localhost | 5050 | `${PGADMIN_EMAIL}` | `${PGADMIN_PASSWORD}` | PostgreSQL 可视化 | 否 |
| Redis | localhost | 6379 | 无 | 无 | 实时缓存、幂等键、临时状态 | 是 |
| Kafka | localhost | 9092 | 无 | 无 | 弹幕事件、库存事件、内部指令流 | 是 |
| Kafka UI | localhost | 9093 | 无 | 无 | Kafka 可视化 | 否 |
| MinIO API | localhost | 8900 | `${MINIO_ACCESS_KEY}` | `${MINIO_SECRET_KEY}` | 可选：大文件、长手卡、报告附件卸载 | P2 |
| MinIO 控制台 | localhost | 8901 | `${MINIO_ACCESS_KEY}` | `${MINIO_SECRET_KEY}` | 可选：对象存储可视化 | P2 |
| Milvus gRPC | localhost | 19530 | 无 | 无 | 可选：替代 pgvector 的向量检索实验 | P2 |
| MySQL | localhost | 3306 | `${MYSQL_USER}` | `${MYSQL_PASSWORD}` | 可选：对比或迁移实验，不作为主库 | 否 |

基础设施注意事项：

- PostgreSQL 首次启用向量检索前，需要执行 `CREATE EXTENSION IF NOT EXISTS vector;`。
- Kafka KRaft 模式首次使用消费者组前，可能需要手动创建 `__consumer_offsets` topic。
- MinIO API 端口为 `8900`，不要使用 Windows 保留端口段中的 `9000`。
- 所有容器均为手动启动，开发前需确认对应 Docker Compose 服务已运行。

## 11. 术语表

| 术语 | 释义 |
| :--- | :--- |
| 手卡 | 为主播准备的单个商品讲解提示卡，包含卖点、话术、价格建议等 |
| 货盘 | 当前直播间中所有待讲解商品的集合 |
| 切品 | 从当前正在讲解的商品切换到另一个商品 |
| 排品 | 安排商品的讲解顺序 |
| 引流款 | 低价高性价比商品，用于吸引观众进入直播间 |
| 利润款 | 高毛利商品，用于创造核心营收 |
| 氛围款 | 用于活跃直播间气氛、提升互动的商品 |
| 信任分 | 衡量主播对 Agent 建议可靠度认可程度的数值，范围 0.0-1.0 |
| 副屏/提词器 | 主播辅助显示屏，用于展示 Agent 推送的实时建议 |
| Harness | 外层工程化骨架，用确定性代码约束大模型的不确定性行为 |
| Reducer | 借鉴 Redux 的纯函数状态管理模式，拦截 LLM 输出 Action 并执行确定性状态变更 |
| Decision Trace | 记录 Agent 建议、主播反馈、工具调用和业务结果的审计轨迹 |
| Hard-gate | 高危操作必须等待主播确认后才能执行的阻塞式审批 |
| Fail-Closed | 系统在不确定是否安全时默认拒绝执行 |
