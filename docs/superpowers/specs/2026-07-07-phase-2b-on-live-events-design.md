# Phase 2B 基础播中事件闭环设计

## 背景与目标

Phase 2A 已经完成播前业务 MVP：样例货盘、排品、商品手卡、模拟建播和审计。Phase 2B 开始进入播中阶段，但只建设最小可控闭环：处理本地模拟售罄事件，验证生命周期边界、Reducer 状态变更、备选商品推荐、主播提示和审计链路。

本阶段不接 LLM、不做 Web 前端、不接真实淘宝 API、不启动长期 Kafka consumer。Kafka 仍作为后续入口保留，本阶段用确定性事件模型先验证业务链路。

## 设计边界

- 事件入口：`InventoryEvent` 只支持 `sold_out`，未知事件类型 fail-closed。
- 生命周期：播中事件只能在 `ON_LIVE` 状态处理，`PRE_LIVE` 和 `POST_LIVE` 均拒绝。
- 状态更新：售罄下架和切换商品继续通过 Reducer，业务流服务不得直接修改状态。
- 推荐策略：备选商品只从仍上架且库存大于 0 的商品里选择，优先转化率和库存。
- 主播提示：使用确定性模板，不编造平台数据，不替主播做真实承诺。
- 审计：售罄处理、备选推荐、主播提示都写入 `tool_call_audit`，并共用同一个 `trace_id`。

## 工具注册

Phase 2B 新增三个播中工具：

- `handle_sold_out_event`：处理售罄事件，高风险、自动执行、需要幂等键。
- `recommend_backup_product`：推荐备选商品，中风险、自动执行。
- `generate_on_live_prompt`：生成主播提示，低风险、自动执行。

这些工具只允许在 `ON_LIVE` 生命周期使用。

## 验收标准

- 非 `ON_LIVE` 状态处理售罄事件会被拒绝。
- 售罄商品被 Reducer 设置为库存 0 且下架。
- 存在备选商品时，系统推荐备选并切换当前讲解商品。
- 不存在备选商品时，系统不切换商品，提示人工接管。
- CLI 能演示完整售罄闭环，并输出 `trace_id` 和审计 `audit_id`。
- 阶段执行记录必须包含测试结果、CLI 结果、问题修复和后续迭代建议。
