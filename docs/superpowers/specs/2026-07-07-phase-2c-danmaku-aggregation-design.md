# Phase 2C 基础弹幕聚合与参考回复设计

## 背景与目标

Phase 2B 已经完成基础播中售罄事件闭环。Phase 2C 继续留在播中阶段，但把能力从库存事件扩展到弹幕问题处理：用本地脱敏弹幕批次跑通 `ON_LIVE -> 5 秒窗口聚合 -> 同类问题合并 -> 确定性参考回复 -> 审计 -> CLI 演示`。

本阶段不接 LLM、不接 Kafka consumer、不做 Web 前端、不接真实平台 API。弹幕回复只作为主播参考话术，不自动发送给观众。

## 设计边界

- 事件入口：`DanmakuEvent` 只接收脱敏 `viewer_id`、弹幕内容、事件时间、直播间和 `trace_id`。
- 生命周期：弹幕聚合只能在 `ON_LIVE` 状态处理，`PRE_LIVE` 和 `POST_LIVE` 均拒绝。
- 聚合窗口：默认 5 秒，按价格、库存、优惠、物流、使用方法、售后和通用问题做确定性分类。
- 参考回复：使用模板生成，必须提示以页面、平台或主播确认信息为准，不编造价格、库存或售后承诺。
- 状态更新：本阶段不调用 Reducer，不改变商品状态。
- 审计：聚合工具和回复工具都写入 `tool_call_audit`，并共用同一个 `trace_id`。

## 工具注册

Phase 2C 新增两个播中工具：

- `aggregate_danmaku_questions`：聚合同类弹幕问题，低风险、自动执行。
- `generate_danmaku_reply`：生成主播参考回复，中风险、soft-gate。

这些工具只允许在 `ON_LIVE` 生命周期使用。

## 验收标准

- 非 `ON_LIVE` 状态处理弹幕会被拒绝。
- 空弹幕内容、空直播间、空 `trace_id` 被模型校验拒绝。
- 5 秒窗口内同类问题能被合并计数。
- 价格、库存、优惠等高频问题能生成稳定参考回复。
- 未命中分类的问题归入通用问题，并要求人工复核。
- CLI 能演示完整弹幕聚合闭环，并输出 `trace_id` 和审计 `audit_id`。
- 阶段执行记录包含测试结果、CLI 结果、问题修复和后续迭代建议。
